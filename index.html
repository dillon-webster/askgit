Software Development Presentation

Ship Anywhere with Docker

Package your app and everything it needs into a container that runs the same on any machine — your laptop, your teammate's laptop, or a server in the cloud.

It works on my machine.

You build something, it runs perfectly. Someone else clones it and gets errors — wrong Python version, missing library, different operating system. This is so common it became a meme. Docker kills the meme.

Docker bundles your app with its exact runtime environment — the OS layer, dependencies, config, all of it — into a single unit called a container. If it runs on your machine, it runs on anyone's.

Concepts
Key Concepts

Four things to know.

Docker has a small vocabulary. Learn these four terms and the rest makes sense.

▢

Image

A read-only snapshot of your app and its environment. Think of it as the blueprint — shareable, versioned, stored on Docker Hub.

▶

Container

A live, running instance of an image. Start, stop, pause, and delete containers without ever touching the image itself.

📜

Dockerfile

A plain text file with instructions for building an image. Defines the base OS, installs dependencies, copies your code.

☁

Docker Hub

A public registry for sharing images. Like GitHub, but for Docker images. Pull official images for Postgres, Node, Python, and more.

When to use Docker
Use Cases

When does Docker actually help?

Not every project needs Docker, but it shines in these situations:

  • You want others to run your app with zero setup beyond Docker itself
  • Your app has complex dependencies — databases, ML models, multiple services
  • You're deploying to a server and need consistent environments
  • You want to isolate an app from the rest of your system

Dockerfile
Under the Hood

How a Dockerfile works.

A Dockerfile is a recipe. Each line is an instruction. Docker executes them top to bottom and produces a layered image. Here's a simplified version of the one powering askgit:

Dockerfile
FROM ollama/ollama # start from an existing base image

RUN apt-get install -y python3 python3-pip # install system dependencies

COPY server/ ./server/ # copy your app code in

RUN ollama pull llama3.2:3b # bake the AI model into the image

ENTRYPOINT ["./entrypoint.sh"] # what runs when the container starts

Each instruction creates a layer in the image. Docker caches layers, so rebuilds are fast — it only re-runs the steps that changed.

try-it.sh
Hands-On

Try it yourself.

Run askgit on your own machine in five steps. Only prerequisite: Docker Desktop.

01

Install Docker Desktop

One time

Download from docker.com and run the installer. Once it's open you'll see a whale icon in your menu bar — that means Docker is running.

02

Open your terminal and run one command

Pick a folder you can find again first — this downloads a chat.sh file into your current directory, and that's where you'll re-run it from later. Your home folder is a safe bet:

Terminal
~ $ cd ~
~ $ curl -s https://raw.githubusercontent.com/dillon-webster/askgit/main/chat.sh -o chat.sh && chmod +x chat.sh && ./chat.sh
03

Wait for the pull

~5 minutes

The first run downloads about 5GB — the entire runtime, server, and AI model bundled together. You'll see progress bars. Go grab a coffee. Every run after this is instant because Docker caches the image.

04

Start chatting

You're live

When you see a You: prompt, the container is up and the model is loaded. Ask it anything about Git — commands, concepts, workflows, debugging. Everything runs locally. Nothing leaves your machine. Press Ctrl+C to exit the chat (the container keeps running in the background).

05

Re-running it later

Instant

No re-downloading — the image is cached for good. Just go back to the folder you ran it in and run the script again:

Terminal
~ $ cd ~ # wherever you ran it the first time
~ $ ./chat.sh
You:

chat.sh reuses the existing container if it's there, or restarts it in a second or two — you're back at the You: prompt almost instantly.

Get Started — Docker Compose
Do It Yourself

A real example: my school project.

Real apps aren't one thing. My school project is three pieces that have to run together:

  • A React + Vite frontend
  • A FastAPI (Python) backend
  • A Postgres database

Wiring those up by hand — right Python version, right Node version, install Postgres, make them talk to each other — is exactly the "works on my machine" nightmare. Docker Compose describes all three in one file and starts them with one command.

01

Each piece gets its own Dockerfile

Same recipe idea as before, one per service. The backend's is tiny:

backend/Dockerfile
FROM python:3.14.3 # Python runtime
COPY . . # copy the backend code in
RUN pip install -r requirements.txt # install deps
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

The frontend has a matching one: FROM node:18 → npm install → npm run preview.

02

One docker-compose.yml ties them together

This declares all three services, their ports, and how they depend on each other. build: means "build from that folder's Dockerfile"; image: means "pull this one from Docker Hub":

docker-compose.yml
services:
  backend:
    build: ./backend/
    ports: ["8000:8000"]
    depends_on: [db] # wait for the database
    environment:
      DATABASE_URL: postgresql://dillon:postgres@db:5432/school-api
  frontend:
    build: ./frontend/
    ports: ["4173:4173"]
    depends_on: [backend]
  db:
    image: postgres:15 # pulled straight from Docker Hub
    volumes: [postgres_data:/var/lib/postgresql/data] # data survives restarts
03

Start the whole stack

One command

From the project root — this builds the two images, pulls Postgres, creates a private network so they can find each other, and starts all three:

Terminal
~/school $ docker compose up --build
[+] Running 3/3 ✔ Container school-db-1 Started ✔ Container school-backend-1 Started ✔ Container school-frontend-1 Started
04

It's all running

You're live

Three services, three ports, talking to each other on a private network — and the only thing anyone installed was Docker:

Service
Where it lives
frontend
localhost:4173
backend
localhost:8000
db (postgres)
internal — backend reaches it at db:5432

The key trick: the backend finds the database at the hostname db — the service name. Docker Compose gives every service a name on a shared network, so containers talk to each other without hardcoded IPs.

Compose cheat-sheet

The handful you'll actually use:

Command
What it does
